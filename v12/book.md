`v12` では、ucontext API を使ってユーザー空間でのコンテキストスイッチを実装します。
Round-Robin と優先度スケジューリングの 2 つの方式を比較し、「スケジューラとは何をしているのか」を体験します。

## 概要

OS のスケジューラは、複数のタスクに CPU 時間を分配します。
その中核にあるのが**コンテキストスイッチ** --- あるタスクの CPU 状態（レジスタ、スタックポインタ）を丸ごと保存し、別のタスクの状態を復元する操作です。

v4 では `push` / `pop` で個別のレジスタを退避・復元しました。
コンテキストスイッチはその究極形で、全レジスタとスタックを一括で切り替えます。

| v4 (push/pop) | v12 (ucontext) | 違い |
|----------------|-----------------|------|
| `push rbx` | `swapcontext(&from, &to)` | 1 レジスタ → 全レジスタ |
| 手動でスタック操作 | OS がスタック管理 | 手動 → 自動 |
| 1 関数内の退避 | タスク間の切り替え | 関数内 → タスク間 |

## コンテキストとは何か

CPU の「現在の状態」の全体をコンテキストと呼びます。具体的には:

- **汎用レジスタ**: rax, rbx, rcx, rdx, rsi, rdi, r8-r15 など（v1-v4 で操作した値の格納場所）
- **スタックポインタ (rsp)**: 現在のスタック位置（v3 で学んだ call/ret の基盤）
- **インストラクションポインタ (rip)**: 次に実行する命令のアドレス
- **フラグレジスタ (RFLAGS)**: 演算結果のフラグ（v2 で使った CF, ZF など）

コンテキストスイッチは、これらを丸ごとメモリに退避し、別のタスク用の値をレジスタに書き戻す操作です。

![コンテキストスイッチ: レジスタの退避と復元](/images/v12/context-switch.drawio.svg)

```
タスク A 実行中                    タスク B 実行中
┌──────────────┐                  ┌──────────────┐
│ rax = 42     │  ── 退避 ──→     │ rax = 99     │
│ rsp = 0x7000 │  ← 復元 ──      │ rsp = 0x8000 │
│ rip = 0x401  │                  │ rip = 0x502  │
└──────────────┘                  └──────────────┘
```

## ucontext API

ucontext は POSIX が提供するユーザー空間コンテキストスイッチ API です。
3 つの関数で構成されます:

| 関数 | 役割 | v4 との対応 |
|------|------|-------------|
| `getcontext(&ctx)` | 現在の CPU 状態を ctx に保存 | 全レジスタの push |
| `makecontext(&ctx, func, ...)` | ctx に新しいエントリポイントを設定 | call 先の設定 |
| `swapcontext(&old, &new)` | old に保存 → new を復元 | push all + pop all + jmp |

### タスク生成の流れ

```c
ucontext_t ctx;

/* ステップ 1: 現在の状態をテンプレートとしてコピー */
getcontext(&ctx);

/* ステップ 2: 専用スタックを割り当て */
ctx.uc_stack.ss_sp   = stack_memory;  /* スタック領域の先頭 */
ctx.uc_stack.ss_size = 65536;         /* 64KB */

/* ステップ 3: エントリポイントを登録 */
makecontext(&ctx, task_function, 0);
```

`getcontext` が全レジスタを保存し、そこにスタックとエントリポイントを上書きすることで「新しいタスク」を作ります。

### コンテキストスイッチの実行

```c
/* タスク A の状態を保存し、タスク B の状態を復元 */
swapcontext(&task_a.ctx, &task_b.ctx);
/* ↑ タスク A は次に復元されるとき、ここから再開する */
```

`swapcontext` は内部で「全レジスタの push → 別タスクの全レジスタの pop → jmp」に相当する処理を行います。

## Round-Robin スケジューラ

Round-Robin (RR) は最もシンプルなスケジューリング方式です。
タスクを登録順に 1 回ずつ実行し、全タスクを一巡したら最初に戻ります。

### 実行の流れ

| ステップ | スケジューラの動作 | 実行タスク | 出力 |
|----------|--------------------|------------|------|
| 1 | タスク A を選択、swapcontext | A | A:1 |
| 2 | A が yield → スケジューラに戻る | — | — |
| 3 | タスク B を選択、swapcontext | B | B:1 |
| 4 | B が yield → スケジューラに戻る | — | — |
| 5 | タスク C を選択、swapcontext | C | C:1 |
| 6 | C が yield → スケジューラに戻る | — | — |
| 7 | タスク A を選択（2 周目）| A | A:2 |
| ... | 以下同様 | ... | ... |
| 18 | 全タスク FINISHED → 終了 | — | — |

### yield の仕組み

各タスクは `yield()` を呼んでスケジューラに制御を返します。

```
タスク A 内:
  printf("A:1");
  yield();          ← swapcontext(&task_a, &scheduler)
  // ↑ 次に A が選ばれると、ここから再開
  printf("A:2");
  yield();
```

yield は「自分の状態を保存してスケジューラに戻る」だけです。
次にスケジューラが自分を選べば、yield の直後から再開します。

## 優先度スケジューラ

優先度スケジューラは、毎回最も優先度の高い READY タスクを選びます。
高優先度タスクが完了するまで、低優先度タスクは実行されません。

### 実行の流れ

タスク設定: HIGH (優先度 3), MED (優先度 2), LOW (優先度 1)

| ステップ | pick_highest() の結果 | 実行タスク | 出力 |
|----------|-----------------------|------------|------|
| 1 | HIGH (prio=3) | HIGH | HIGH:1 |
| 2 | HIGH (prio=3) | HIGH | HIGH:2 |
| 3 | HIGH 完了 → MED (prio=2) | MED | MED:1 |
| 4 | MED (prio=2) | MED | MED:2 |
| 5 | MED 完了 → LOW (prio=1) | LOW | LOW:1 |
| 6 | LOW (prio=1) | LOW | LOW:2 |
| 7 | 全タスク完了 → 終了 | — | — |

### RR との比較

```
Round-Robin:     A:1 → B:1 → C:1 → A:2 → B:2 → C:2 → A:3 → B:3 → C:3
優先度:          HIGH:1 → HIGH:2 → MED:1 → MED:2 → LOW:1 → LOW:2
```

RR は「公平性」を重視し、優先度は「重要度」を重視します。
実際の OS カーネルは両方を組み合わせた方式を使います（Linux の CFS など）。

### 飢餓問題

純粋な優先度スケジューラには**飢餓 (starvation)** の問題があります。
高優先度タスクが常に READY であれば、低優先度タスクは永遠に実行されません。

実際のカーネルでは:
- **エイジング**: 長時間待たされたタスクの優先度を動的に上げる
- **タイムスライス**: 高優先度でも一定時間で強制的に切り替える
- **優先度逆転防止**: 低優先度タスクがロックを保持している場合、一時的に優先度を引き上げる

## PlayStation との関連

- **コンテキストスイッチ**: PS4/PS5 のカーネルは FreeBSD ベースで、ゲームスレッドの切り替えにコンテキストスイッチを使います。ゲームでは 16.6ms (60fps) 以内にフレームを完成させる必要があり、スケジューリングの効率が直接フレームレートに影響します
- **コアアフィニティ**: ゲームエンジンは特定のスレッドを特定の CPU コアに固定（pin）します。レンダリングスレッドをコア 0 に、物理演算をコア 1 にといった配置で、コンテキストスイッチのオーバーヘッドを最小化します。これは v12 の「タスクをどう配分するか」という問題そのものです
