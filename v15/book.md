`v15` では、Memory-Mapped I/O（MMIO）によるデバイスアクセスをシミュレーションします。
特定のメモリアドレスへの load/store がデバイスへの命令になる仕組みを、mmap と volatile ポインタで再現します。

## 概要

CPU がデバイス（ディスクコントローラ、ネットワークカード、GPU など）と通信するには、
何らかのインターフェースが必要です。代表的な方式が 2 つあります。

| 方式 | 仕組み | 例 |
|------|--------|----|
| Port I/O | 専用の `in`/`out` 命令でデバイスにアクセス | x86 の I/O ポート |
| Memory-Mapped I/O | 通常の `mov` 命令（load/store）でアクセス | ARM、PCIe デバイス、GPU |

MMIO では、デバイスのレジスタが物理メモリ空間の特定アドレスにマッピングされます。
CPU から見ると普通のメモリアクセスですが、メモリコントローラがアクセス先を判別し、
デバイスにリクエストを転送します。

```
CPU                メモリコントローラ          デバイス
 |                      |                      |
 |-- store 0xA000 -->   |                      |
 |                      |-- 「0xA000はデバイス」-->|
 |                      |                      |-- コマンド実行
 |                      |<-- ステータス応答 ---|
 |<-- load 0xA004 ---   |                      |
```

## v9 との関係: なぜ volatile が必要だったか

v9 では `volatile` キーワードを学びました。「コンパイラが最適化でアクセスを省略するのを防ぐ」という説明でしたが、
それが必要になる最も典型的な場面が MMIO です。

```c
/* v9: volatile の基本 */
volatile int device_reg = 0;
device_reg = 10;   /* 省略されてはいけない */
device_reg = 20;   /* これも省略されてはいけない */

/* v15: 実際のデバイスポーリングパターン */
volatile uint32_t *status = (volatile uint32_t *)0xA004;
while (*status != DONE) {   /* 毎回読み直す必要がある */
    /* wait */
}
```

volatile がなければ、コンパイラは `*status` の値を一度レジスタに読み込んだ後、
ループ内で再読み込みを省略できます。デバイスが STATUS を BUSY から DONE に更新しても、
CPU は古い値を見続けて永久にループすることになります。

## レジスタレイアウト

MMIO デバイスには複数のレジスタがあり、ベースアドレスからのオフセットで区別します。

| オフセット | レジスタ | 方向 | 用途 |
|-----------|---------|------|------|
| `+0x00` | COMMAND_REG | Write | デバイスへのコマンド送信 |
| `+0x04` | STATUS_REG | Read | デバイスの状態確認 |
| `+0x08` | DATA_REG | R/W | データの読み書き |

例えばベースアドレスが `0xA000` なら:
- `0xA000` に書き込む = コマンド送信
- `0xA004` を読み出す = ステータス確認
- `0xA008` を読み出す = データ取得

このレイアウトはデバイスのデータシート（仕様書）に記載されます。
ドライバ開発者はデータシートを読んで、どのオフセットに何を書けばよいかを知ります。

## mmio_sim --- コマンド送信とステータスポーリング

MMIO デバイスとの典型的なやりとりは以下の手順です。

| ステップ | 操作 | C コード | 意味 |
|----------|------|----------|------|
| 1 | MMIO 領域確保 | `mmap(NULL, size, ...)` | デバイスレジスタ空間のマッピング |
| 2 | コマンド送信 | `*command_reg = CMD_READ` | COMMAND_REG への store |
| 3 | デバイス処理 | (デバイスが STATUS を更新) | ハードウェア側の処理 |
| 4 | ポーリング | `while (*status_reg != DONE)` | STATUS_REG の繰り返し load |
| 5 | データ読み出し | `data = *data_reg` | DATA_REG からの load |

### ポーリングと割り込み

このプログラムではステータスをポーリング（繰り返し確認）して待っています。
シンプルですが CPU 時間を浪費します。実際のデバイスドライバでは割り込み（interrupt）を使うことが多いです。

| 方式 | 特徴 | 用途 |
|------|------|------|
| ポーリング | CPU がループで確認。シンプルだが CPU を占有 | 短時間で完了する操作、組み込みシステム |
| 割り込み | デバイスが完了時に CPU に通知。効率的 | ディスク I/O、ネットワークパケット受信 |

## mmio_led --- LED コントローラのシミュレーション

組み込みシステムの LED 制御は MMIO の最もシンプルな例です。
レジスタの各ビットが 1 つの LED に対応します。

| ビット | LED | 値 1 | 値 0 |
|--------|-----|------|------|
| ビット 3 | LED3 | 点灯 | 消灯 |
| ビット 2 | LED2 | 点灯 | 消灯 |
| ビット 1 | LED1 | 点灯 | 消灯 |
| ビット 0 | LED0 | 点灯 | 消灯 |

パターン例:
- `0xF` (= `1111b`): 全 LED 点灯 → `[*][*][*][*]`
- `0xA` (= `1010b`): 交互点灯 → `[*][ ][*][ ]`
- `0x0` (= `0000b`): 全 LED 消灯 → `[ ][ ][ ][ ]`

### 読み戻し（Read-back）

MMIO では、書き込んだ値をそのまま信用せず、読み戻して確認するのが安全です。
デバイスによっては書き込んだ値と読み出す値が異なることがあります。

```c
/* 書き込み */
*control_reg = 0xF;

/* 読み戻して確認（実デバイスでは値が異なる可能性がある） */
uint32_t actual = *status_reg;
```

例えば、ハードウェアの制約で一部の LED が故障している場合、
`0xF` を書き込んでも `0x7` しか読み戻せないかもしれません。
